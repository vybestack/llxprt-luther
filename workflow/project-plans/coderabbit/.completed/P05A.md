## Semantic Verification

### Artifact store canonical/current and immutable history writes

Verified in `src/engine/executors/pr_followup_artifacts.rs`:

- `write_json_artifact` allocates an `ArtifactSequenceMetadata`, computes canonical and history paths, injects binding/common/failure fields, serializes the JSON object, writes the history file, then writes the canonical current file.
- `ArtifactWriter::canonical_path` writes under `<root>/pr-followup/current/<run_id>/<owner>/<repo>/<pr_number>/<family>.json`.
- `ArtifactWriter::history_path` writes under `<root>/pr-followup/history/<run_id>/<owner>/<repo>/<pr_number>/<family>/<artifact_sequence>-<write_sequence>-<producer_step_id>.json`.
- `HistoryMetadata` includes canonical path, history path, artifact family, `is_canonical`, and write timestamp in every stored payload.

### Atomic rename and same-directory temp files

Verified in `atomic_write`: it creates the destination parent directory, creates a temp path as `parent.join(format!(".<filename>.<uuid>.tmp"))`, writes bytes to that temp path, and renames the temp path into the final destination. Because the temp path is joined to the destination parent, temp files are in the same directory as the target for rename semantics.

### Sequence contract

Verified by tests and code inspection:

- Global `artifact_sequence` is recovered from all accepted same-run candidate snapshots and incremented globally.
- Per-family `write_sequence` is recovered from `max_write_sequence_by_family` and incremented per artifact family.
- Global `failure_sequence` is recovered across failure-producing artifacts and incremented globally.
- The test `artifact_store_allocates_global_artifact_and_failure_sequences_with_per_family_write_sequences_for_interleaved_families` confirms interleaved writes allocate artifact sequences `1,2,3`, per-family write sequences `1,1,2`, and failure sequences `1,2,None`.

### Binding validation

Verified in `validate_artifact_value`, `binding_from_value`, and `ArtifactWriter::validate_binding`:

- Artifacts are rejected when required binding fields are absent or mismatched.
- `history_metadata` and its required fields are required.

- Current-head artifacts with stale/unbound/mismatched binding are rejected rather than consumed.
