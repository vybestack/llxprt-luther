```
impl StepExecutor for ShellExecutor {
    fn execute(
        &self,
        context: &mut StepContext,
        params: &serde_json::Value,
    ) -> Result<StepOutcome, EngineError> {
        // Extract "command" from params JSON
        let command_template = params
            .get("command")
            .and_then(|v| v.as_str())
            .ok_or_else(|| EngineError::StepExecutionError {
                step_id: "shell".to_string(),
                message: "Missing 'command' parameter for shell executor".to_string(),
            })?;

        // Interpolate command string using context
        let interpolated_command = interpolate_string(command_template, context);

        // Run via sh -c
        let mut cmd = Command::new("sh");
        cmd.arg("-c").arg(&interpolated_command);
        cmd.current_dir(context.work_dir());
        cmd.stdout(Stdio::piped());
        cmd.stderr(Stdio::piped());

        let output = match cmd.output() {
            Ok(output) => output,
            Err(e) => {
                return Err(EngineError::StepExecutionError {
                    step_id: "shell".to_string(),
                    message: format!("Failed to spawn command: {e}"),
                });
            }
        };

        // Capture stdout and stderr into context
        let stdout_str = String::from_utf8_lossy(&output.stdout).to_string();
        let stderr_str = String::from_utf8_lossy(&output.stderr).to_string();
        let exit_code = output.status.code();

        context.set("stdout", &stdout_str);
```

## 5. SCHEMA COMPATIBILITY ANALYSIS

### 5.1 Two TransitionDef structs confirmed
**Location 1:** `src/workflow/schema.rs` (line 70)
**Location 2:** `src/engine/transition.rs` (line 55)

Both structs have identical fields currently:
- `from: String`
- `to: String`
- `condition: Option<String>`

**Missing:** `max_iterations: Option<u32>` in both locations

**Status:** NON-BLOCKING - This is expected per the plan (needs to be added during implementation)

### 5.2 StepContext storage model
**Current:** `variables: HashMap<String, String>` (flat key-value)

**Expected for namespaced storage:** Keys stored as `step_id.variable_name`

**Issue:** HashMap doesn't preserve insertion order. For "most-recent-first" unnamespaced lookups, need to track step execution order.

**Status:** NON-BLOCKING - Plan already addresses this (recommendation to use Vec for step order tracking)

### 5.3 StateSnapshot structure
**Current:** `loop_count: u32` (single global counter)

**Expected:** Add `edge_loop_counts: HashMap<String, u32>` keyed by `"from:to"`

**SQLite schema:** Current schema has `loop_count INTEGER` column. Per-edge counts should go in `context` JSON blob to avoid schema migration.

**Status:** NON-BLOCKING - Plan already addresses this

### 5.4 Checkpoint schema
SQLite table from `src/persistence/checkpoint.rs`:
```rust
CREATE TABLE IF NOT EXISTS checkpoints (
    run_id TEXT NOT NULL,
    step_id TEXT NOT NULL,
    retry_count INTEGER NOT NULL DEFAULT 0,
    loop_count INTEGER NOT NULL DEFAULT 0,

    status TEXT NOT NULL DEFAULT 'running',
```
