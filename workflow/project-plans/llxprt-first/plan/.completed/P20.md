## Summary

CLI Production Wiring phase completed successfully. The `luther-workflow run` command now resolves workflow types and configs from the production `config/` directory instead of the hardcoded `tests/fixtures` path.

## Changes Made

### Modified Files

1. **src/cli/mod.rs**
   - Added `config_dir: Option<PathBuf>` field to `RunArgs` struct
   - Added `#[arg(long, value_name = "DIR")]` attribute
   - Added documentation: "Directory containing workflows/ and workflow-configs/ subdirectories"
   - Updated test to include new field

2. **src/main.rs**
   - Replaced hardcoded `fixture_root = PathBuf::from("tests/fixtures")` with:
     ```rust
     let config_root = args.config_dir
         .clone()
         .unwrap_or_else(|| PathBuf::from("config"));
     ```
   - Updated all calls to `resolve_workflow_type()`, `resolve_workflow_config()`, and `resolve_workflow()` to use `config_root`

3. **src/workflow/config_loader.rs**
   - Modified `resolve_workflow_type()` to check production layout first:
     - `{root}/workflows/{id}.{ext}` (production)
     - `{root}/workflows/valid/{id}.{ext}` (test fixtures, fallback)
   - Modified `resolve_workflow_config()` with same fallback pattern for `workflow-configs/`
   - Both functions support TOML and JSON in both layouts
   - Updated `resolve_workflow()` parameter name from `fixture_root` to `root`

### Created Files

4. **tests/cli_config_resolution_integration.rs**
   - 8 comprehensive tests:

     - `test_resolve_workflow_type_from_valid_subdir`: Test fixture fallback
     - `test_resolve_workflow_config_from_config_dir`: Config production layout
     - `test_resolve_production_workflow_from_config`: Actual config/ directory resolution
     - `test_flat_layout_takes_precedence`: Verifies production layout priority
     - `test_resolve_json_from_config_dir`: JSON production layout
     - `test_resolve_json_from_valid_subdir`: JSON test fixture fallback
     - `test_existing_test_fixtures_resolve`: Backward compatibility verification

## Key Design Decisions

1. **Production layout first**: Flat `workflows/` and `workflow-configs/` directories are checked first
2. **Test fixture fallback**: `valid/` subdirectory is fallback for backward compatibility
3. **JSON support**: Both TOML and JSON are supported in both layouts
4. **No domain logic**: Resolution is purely file-system based, no knowledge of specific workflows
5. **Default to `config/`**: Production default is relative to cwd, not compiled in
